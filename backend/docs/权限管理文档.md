# SmartSpar BikeHub 权限管理文档

## 概述

SmartSpar BikeHub 系统实现了基于角色的访问控制（RBAC）权限管理体系，确保不同用户只能访问其权限范围内的资源和服务。

## 用户角色定义

### 1. 管理员 (admin)
- **权限范围**：拥有系统最高权限，可访问所有功能
- **主要职责**：系统管理、用户管理、数据维护
- **权限级别**：最高

### 2. 调度员 (dispatcher)
- **权限范围**：负责调度任务的创建和管理，查看预测结果和需求分析
- **主要职责**：调度任务分配、需求数据分析、预测结果查看
- **权限级别**：高

### 3. 运维员 (operator)
- **权限范围**：处理分配给自己的调度任务，查看单车历史数据
- **主要职责**：调度任务执行、单车历史数据查询
- **权限级别**：中

### 4. 普通用户 (user)
- **权限范围**：基础功能，可创建需求数据，查看个人信息
- **主要职责**：提交需求、使用基础功能
- **权限级别**：低

## API 接口权限详情

### 1. 站点管理接口 (`/api/stations`)
- **GET `/stations`** - 查看站点列表：所有认证用户
- **POST `/stations`** - 创建站点：仅管理员
- **GET `/stations/{id}`** - 查看单个站点：所有认证用户
- **PUT `/stations/{id}`** - 更新站点：仅管理员
- **DELETE `/stations/{id}`** - 删除站点：仅管理员

### 2. 需求数据接口 (`/api/demand-data`)
- **GET `/demand-data`** - 查看需求数据：管理员、调度员
- **POST `/demand-data`** - 创建需求数据：所有角色
- **GET `/demand-data/{id}`** - 查看单个需求：管理员、调度员
- **PUT `/demand-data/{id}`** - 更新需求数据：管理员、调度员
- **DELETE `/demand-data/{id}`** - 删除需求数据：管理员、调度员
- **GET `/demand-data/statistics`** - 需求统计：管理员、调度员
- **POST `/demand-data/import`** - 批量导入：管理员、调度员

### 3. 预测结果接口 (`/api/predictions`)
- **GET `/predictions`** - 查看预测结果：调度员、管理员
- **POST `/predictions`** - 创建预测结果：调度员、管理员
- **GET `/predictions/station/{station_id}`** - 站点预测：调度员、管理员
- **GET `/predictions/dashboard`** - 预测看板：调度员、管理员

### 4. 单车历史数据接口 (`/api/bike-history`)
- **GET `/bike-history`** - 查看历史数据：调度员、运维员
- **POST `/bike-history`** - 创建历史记录：调度员、运维员
- **GET `/bike-history/{id}`** - 查看单个记录：调度员、运维员
- **PUT `/bike-history/{id}`** - 更新历史记录：调度员、运维员
- **DELETE `/bike-history/{id}`** - 删除历史记录：调度员、运维员
- **GET `/bike-history/station/{station_id}/latest`** - 最新记录：调度员、运维员
- **GET `/bike-history/statistics`** - 历史统计：调度员、运维员
- **POST `/bike-history/batch`** - 批量创建：调度员、运维员

### 5. 用户管理接口 (`/api/users`)
- **GET `/users`** - 查看所有用户：仅管理员
- **POST `/users`** - 创建用户：仅管理员
- **PUT `/users/{id}`** - 更新用户：管理员可更新任意用户，其他用户只能更新自己
- **DELETE `/users/{id}`** - 删除用户：仅管理员
- **GET `/users/profile`** - 查看个人资料：所有认证用户
- **GET `/users/{id}`** - 查看指定用户：管理员可查看任意用户，其他用户只能查看自己

### 6. 调度任务接口 (`/api/dispatch-tasks`)
- **GET `/dispatch-tasks`** - 查看任务：按角色过滤数据
- **POST `/dispatch-tasks`** - 创建任务：调度员、管理员
- **PUT `/dispatch-tasks/{id}`** - 更新任务：按角色限制可更新字段
- **DELETE `/dispatch-tasks/{id}`** - 删除任务：管理员或创建者
- **POST `/dispatch-tasks/{id}/assign`** - 分配任务：调度员、管理员

## 权限装饰器

系统提供了权限装饰器来简化权限控制：

### 基础装饰器
```python
@require_role('admin', 'dispatcher')  # 指定角色列表
@require_admin()                      # 仅管理员
@require_dispatcher_or_admin()       # 调度员和管理员
@require_operator_or_admin()         # 运维员和管理员
@require_dispatcher_operator()       # 调度员和运维员
@require_any_role()                  # 所有认证用户
```

### 使用示例
```python
from app.utils.permissions import require_admin, require_dispatcher_or_admin

@api_bp.route('/stations', methods=['POST'])
@jwt_required()
@require_admin()
def create_station():
    # 仅管理员可访问
    pass

@api_bp.route('/predictions', methods=['GET'])
@jwt_required()
@require_dispatcher_or_admin()
def get_predictions():
    # 调度员和管理员可访问
    pass
```

## 安全措施

### 1. JWT 认证
- 所有 API 接口都需要有效的 JWT Token
- Token 包含用户身份信息和过期时间
- 支持Token刷新机制

### 2. 角色验证
- 每个接口都会验证用户角色权限
- 权限验证在业务逻辑执行前进行
- 未授权访问返回 403 状态码

### 3. 资源过滤
- 根据用户角色自动过滤可见数据
- 确保用户只能访问其权限范围内的资源
- 数据库查询时加入权限过滤条件

### 4. 操作日志
- 记录所有用户操作行为
- 包含操作类型、资源、时间戳等信息
- 便于安全审计和问题追踪

## 错误处理

### 权限相关错误码
- **401 Unauthorized**: 未提供认证令牌或令牌无效
- **403 Forbidden**: 权限不足，无法访问资源
- **404 Not Found**: 资源不存在或无权访问

### 错误响应格式
```json
{
    "error": "权限不足"
}
```

## 最佳实践

### 1. 权限设计原则
- 最小权限原则：用户只获得必需的权限
- 职责分离：不同角色有明确的职责边界
- 默认拒绝：未明确授权的访问默认拒绝

### 2. 开发注意事项
- 使用权限装饰器，避免重复的权限检查代码
- 在数据库查询时考虑权限过滤
- 提供清晰的权限错误信息
- 定期审查和更新权限设置

### 3. 安全建议
- 定期更新JWT密钥
- 实施访问频率限制
- 监控异常访问行为
- 定期进行权限审计
